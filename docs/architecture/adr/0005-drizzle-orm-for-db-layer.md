# 0005 — Drizzle ORM + drizzle-kit for the database layer

**Status:** Accepted
**Date:** 2026-05-24
**Deciders:** Capt. Dan Moi Ng'ong'a (delegated); Claude Code agent

## Context

CLAUDE.md §"Coding conventions" / Backend left the migration tool open: *"Migrations via drizzle-kit or node-pg-migrate (decide in Sprint 1; document the choice)."*

ADR 0004 fixes `@dnca/domain` as the source of truth for entity types. Whatever DB layer we choose must (a) be TypeScript-first so the storage schema can be aligned mechanically with the domain types, (b) not block the use of raw SQL for the parts that aren't tables (RLS policies, audit triggers, custom functions), (c) produce migrations that are reviewable and replayable in CI, and (d) yield a type-safe query interface so backend handlers cannot silently send malformed queries to production.

The candidates:

1. **Drizzle ORM + drizzle-kit** — TypeScript schema, codegen-derived migrations, lightweight query builder, raw-SQL escape hatch via `sql\`...\``.
2. **node-pg-migrate** — pure SQL migrations, no schema-in-TS, no type-safe query layer. Pairs typically with `pg` directly or a hand-rolled query layer.
3. **Prisma** — schema DSL (`schema.prisma`), excellent DX, but a separate generated client and a known weaker story for row-level security and custom Postgres types.
4. **Kysely + Kysely-codegen** — TypeScript query builder over a generated schema; migrations via separate library (e.g. kysely-migrator).

## Decision

Drizzle ORM + drizzle-kit. Schema definitions live in `packages/db/src/schema/`, mirroring the `@dnca/domain` structure (operator, aircraft, pilot, currency, training, document, governance). Migrations are generated by drizzle-kit and committed under `infra/migrations/`.

The bootstrap migration (`0001_initial.sql`) is hand-written because it includes RLS policies, the audit-log immutability trigger, the `updated_at` trigger function, and database role creation — none of which drizzle-kit auto-generates. Subsequent additive migrations (new columns, new tables, additive indexes) will be generated by drizzle-kit and reviewed before commit.

Custom SQL (RLS policies, triggers, functions, role grants) lives in `infra/migrations/` as numbered post-table migrations so the dependency direction is clear: schema first, then policies that reference schema.

Drizzle's `relations()` API is used to encode FK semantics in TypeScript without requiring an ORM-style entity layer — the query builder remains explicit, which is the right trade for an aviation-records system where every query should be reviewable.

## Consequences

- Backend handlers receive `Database` instances and run type-safe queries — a SELECT misnaming a column is a compile error.
- The schema file is reviewable in code; drift between domain types and storage is caught by an integration test (`assert drizzle-inferred type structurally matches @dnca/domain`).
- Migrations are SQL on disk and run by `drizzle-kit push` (dev) / `drizzle-kit migrate` (prod). No DSL-to-SQL translation step at deploy time.
- Custom Postgres features (RLS, triggers, generated columns, jsonb operators, partial indexes) are first-class — they live in raw SQL post-table migrations.
- Drizzle does not auto-detect RLS policy or trigger drift; we own that vigilance via review discipline.

## Alternatives considered

- **node-pg-migrate** — viable but loses the type-safe query layer; we'd be hand-rolling repository methods. The Phase-0 audit notes typing as a hard requirement; this option leaves too much room for query-shape errors.
- **Prisma** — strong DX but RLS support is awkward (no first-class policy DSL); the generated client is a runtime artifact that diverges from the source of truth in `@dnca/domain`.
- **Kysely** — close second; rejected only because drizzle-kit's schema-as-source pattern is a tighter fit with ADR 0004.

## Migration discipline

- Bootstrap migrations: hand-written SQL, numbered `0001_…`, `0002_…`, etc.
- Schema changes: write the Drizzle schema first, run `drizzle-kit generate` to emit the migration, review the SQL, commit both schema and migration.
- RLS / trigger / function changes: hand-written SQL post-table migrations; never auto-generated.
- Migrations are forward-only in production. A reverse migration is a new forward migration.
- Every migration is tested against a Testcontainers Postgres in CI before merge.
